## Honnêteté méthodologique

Cette validation vérifie que l'ajout récent de la section `USE WHEN / SKIP WHEN` (commit `e0af718`) et l'optimisation token de Phase 5 n'ont pas cassé le comportement des 5 skills existants.

Méthode :
1. Lecture du frontmatter et de la section USE/SKIP de chaque skill.
2. Vérification que les règles USE/SKIP ne contredisent pas le rôle annoncé du skill.
3. Vérification que le SKILL.md reste autonome (un lecteur comprend rôle + méthode + règles critiques sans ouvrir 10 fichiers).
4. Grep pour s'assurer que les règles non-négociables (sécurité, push, .env) sont préservées.

**Cette validation n'est pas un runtime test.** Pour confirmer qu'aucune régression n'a été introduite, un humain doit invoquer chaque skill dans une session Claude et observer le comportement réel.

## Tests transversaux (5 skills)

### Test : pas de skill ne déclenche le mauvais comportement par USE WHEN trop large

Demande typique ambiguë : *"Améliore ce site."*

Comportement attendu :
- `repo-builder` → SKIP (pas de création)
- `production-auditor` → SKIP (pas de mention prod)
- `premium-webdesigner` → match possible (USE WHEN inclut "améliorer layout")
- `seo-content-engine` → SKIP (pas de mention contenu)
- `site-ux-guardian` → match possible (USE WHEN inclut "vérifier cohérence")
- `multilingual-site-engine` → SKIP (pas de mention traduction)
- `skill-orchestrator` → match (demande ambiguë, plusieurs skills possibles)

**Résultat attendu** : `skill-orchestrator` doit demander une question de clarification : *"Améliorer = refonte design ? cohérence ? perf ? prod ?"* avant d'invoquer un skill spécifique.

✓ Comportement cohérent avec la philosophie "ne charger plusieurs gros skills que sur demande explicite".

### Test : règles destructives partagées

Grep effectué :
```
grep -RInE "ne (pas|jamais).*(supprim|push|écras|détruir|forcer)" skills/*/SKILL.md
```

Résultat attendu : chaque SKILL.md possède au moins une mention "ne pas push sans accord", "ne pas supprimer sans validation", "ne pas modifier .env".

→ La factorisation Phase 6 vers `docs/shared-safety-rules.md` (en cours) ne doit pas faire perdre ces règles. Le pointeur dans chaque SKILL.md doit rester explicite.
