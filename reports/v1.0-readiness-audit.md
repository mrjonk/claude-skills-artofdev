## Honnêteté méthodologique

Cet audit a été produit par analyse **statique** du repo (lecture des fichiers, mesures, grep ciblés). Il ne contient **pas** de runtime testing : Claude ne peut pas observer le comportement réel d'un skill dans une session live. Les "validations" produites en Phase 2-4 sont donc des **revues de spec + fixtures testables** — un humain (ou Claude dans une session ultérieure) doit confirmer le runtime sur de vrais projets.

C'est précisément pour ça que la cible de cette préparation est **`v1.0.0-rc1`** (release candidate) et **pas** `v1.0.0` final.

## Doublons probables (à valider en Phase 6)

Zones à risque identifiées :

1. **Sécurité de base** : `production-auditor/references/security-checklist.md` ↔ `site-ux-guardian/references/security-sanity-checklist.md` ↔ `repo-builder/references/security-checklist.md`. Les périmètres se chevauchent (ne pas commiter `.env`, pas de secrets en clair, etc.). À fusionner ou pointer vers une référence partagée `docs/shared-safety-rules.md`.
2. **Webdesign / UX** : `premium-webdesigner/references/ux-ui-checklist.md` ↔ `site-ux-guardian/references/webdesign-guidelines.md` ↔ `production-auditor/references/ux-ui-checklist.md`. Trois UX-checklists. Risque réel.
3. **SEO** : `seo-content-engine/references/seo-checklist.md` ↔ `production-auditor/references/seo-content-checklist.md` ↔ `multilingual-site-engine/references/seo-hreflang-guide.md`. Périmètres distincts à confirmer.
4. **Anti-style IA** : règle "remplacer le tiret long quand il sonne IA" présente dans `multilingual-site-engine` et `seo-content-engine`. Acceptable si chaque skill l'applique à son domaine, mais à vérifier qu'il n'y a pas de copie verbatim.
5. **Règles destructives** : "ne pas push sans accord" / "ne pas modifier .env" se répète dans presque chaque SKILL.md. Candidat #1 à factoriser dans `docs/shared-safety-rules.md`.

## Validations manquantes pour atteindre v1.0.0 (final)

Pour passer de **rc1** à **1.0.0 final**, il faut :

1. **Runtime tests** des 7 skills sur de vrais projets — un humain (ou Claude dans une nouvelle session) doit invoquer chaque skill et observer le comportement.
2. **3 stacks réelles** pour `multilingual-site-engine` (PHP, Next.js, WordPress).
3. **15 cas de routage réels** pour `skill-orchestrator`.
4. **Anti-régression observée** des 5 skills existants (les scripts de scan ne suffisent pas — il faut une session Claude qui les active vraiment).
5. **Article éditorial publié** + bannière OG livrée.
6. **Tag `v1.0.0` + release GitHub**.

Aucun de ces 6 points ne peut être validé par moi seul dans cette session.
